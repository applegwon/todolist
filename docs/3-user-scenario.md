## 페르소나 정의

### P-01. 이지수 (대학생, 22세)
- **직업:** 대학교 3학년
- **특징:** 과제, 시험, 동아리 활동 등 다양한 일정을 관리해야 한다. 스마트폰을 주로 사용하며, 단순하고 빠른 할일 등록을 선호한다.
- **Pain Point:** 해야 할 일이 많은데 기한을 놓치는 경우가 잦다.
- **관련 UC:** UC-101~104, UC-401~409

### P-02. 김민준 (직장인, 35세)
- **직업:** IT 회사 프로젝트 매니저
- **특징:** 업무와 개인 일정을 분리하여 관리하고 싶다. 카테고리별 정리를 통해 우선순위를 파악한다. 다크 모드를 선호하며 영어 UI에도 익숙하다.
- **Pain Point:** 업무·개인 할일이 섞여 있어 현황 파악이 어렵다.
- **관련 UC:** UC-201~205, UC-301~306, UC-401~409, UC-501~505

### P-03. 박서연 (직장인, 45세)
- **직업:** 중소기업 경영지원팀 팀장
- **특징:** IT 서비스에 익숙하지 않아 직관적인 UI를 선호한다. 한국어 기본 설정으로 사용하며, 기한이 지난 업무를 빠르게 확인하고 싶다.
- **Pain Point:** 기한이 지난 할일을 별도로 파악하기가 어렵다.
- **관련 UC:** UC-103~104, UC-401~406

### US-01. 신규 사용자 온보딩

**페르소나:** P-01 이지수

**상황 (Context)**
이지수는 학기 초 과제와 시험 일정을 한 곳에 모아 관리하고 싶어 TodoList 앱을 처음 방문했다.

**목표 (Goal)**
회원가입을 완료하고 로그인한 뒤, 첫 번째 할일을 등록한다.

**시나리오 흐름**

1. 이지수가 앱에 접속하면 시스템은 로그인 화면(S-02)을 표시한다.
2. 이지수가 "회원가입" 링크를 클릭하면 시스템은 회원가입 화면(S-01)으로 이동한다.
3. 이지수가 이메일, 비밀번호, 이름을 입력하고 제출한다. (UC-101)
4. 시스템이 이메일 중복 여부를 확인한다. (UC-102)
   - 중복 이메일인 경우: 시스템이 오류 메시지를 표시하고 흐름이 3번으로 돌아간다.
5. 시스템이 계정을 생성하고 로그인 화면(S-02)으로 이동한다.
6. 이지수가 이메일과 비밀번호를 입력하고 로그인한다. (UC-103)
7. 시스템이 JWT를 발급하고 저장된 테마(light)·언어(ko) 설정을 적용한 뒤 할일 목록 화면(S-03)으로 이동한다. (UC-205)
8. 이지수가 할일 추가 버튼을 눌러 할일 등록 화면(S-04)으로 이동한다.
9. 이지수가 제목("중간고사 공부")을 입력하고 저장한다. (UC-401)
10. 시스템이 카테고리를 '기본'으로 자동 적용하고 할일을 저장한 뒤 할일 목록 화면(S-03)으로 돌아간다. (UC-402)

**결과 (Outcome)**
이지수의 계정이 생성되었고, 첫 번째 할일 "중간고사 공부"가 '기본' 카테고리, '미시작' 상태로 등록된다.

### US-02. 카테고리로 할일 관리

**페르소나:** P-02 김민준

**상황 (Context)**
김민준은 업무용 할일과 개인 할일을 분리하여 관리하려 한다. 기존에는 모든 할일이 '기본' 카테고리에 섞여 있었다.

**목표 (Goal)**
"업무" 카테고리를 새로 만들고 업무 관련 할일을 등록한 뒤, 카테고리 필터로 업무 할일만 조회한다.

**시나리오 흐름**

1. 김민준이 로그인 상태에서 카테고리 관리 화면(S-06)으로 이동한다.
2. 김민준이 "카테고리 추가" 버튼을 클릭하고 이름 "업무"를 입력하여 저장한다. (UC-301)
3. 시스템이 동일 사용자 내 이름 중복 여부를 확인한다. (UC-302)
   - 이미 "업무" 카테고리가 존재하는 경우: 오류 메시지를 표시하고 흐름이 2번으로 돌아간다.
4. 시스템이 "업무" 카테고리를 생성하고 목록에 표시한다.
5. 김민준이 할일 등록 화면(S-04)으로 이동하여 제목("주간 보고서 작성"), 카테고리("업무")를 선택하고 저장한다. (UC-401)
6. 시스템이 할일을 저장하고 할일 목록 화면(S-03)으로 돌아간다.
7. 김민준이 할일 목록 화면(S-03)에서 카테고리 필터를 "업무"로 설정한다. (UC-405)
8. 시스템이 "업무" 카테고리에 속한 할일만 필터링하여 표시한다. (UC-404)

**결과 (Outcome)**
"업무" 카테고리가 생성되고, "주간 보고서 작성" 할일이 "업무" 카테고리에 등록된다. 필터 적용 시 업무 할일만 조회된다.

### US-03. 기한 관리

**페르소나:** P-03 박서연

**상황 (Context)**
박서연은 처리가 밀린 업무를 파악하기 위해 기한이 초과된 할일을 한눈에 보고 싶다.

**목표 (Goal)**
시작일·종료일을 설정하여 할일을 등록하고, 기한초과 필터를 사용해 미완료 기한초과 할일만 조회한다.

**시나리오 흐름**

1. 박서연이 로그인 상태에서 할일 등록 화면(S-04)으로 이동한다.
2. 박서연이 제목("거래처 계약서 검토"), 시작일자(2026-05-20), 종료일자(2026-05-22)를 입력하고 저장한다. (UC-401)
3. 시스템이 종료일자가 시작일자보다 이전이 아님을 확인한다. (UC-403)
4. 시스템이 할일을 저장하고 할일 목록 화면(S-03)으로 돌아간다.
5. 오늘 날짜(2026-05-27)가 종료일자(2026-05-22)를 지났으므로, 시스템은 이 할일의 기한초과 여부를 true로 동적 계산한다. (UC-406)
6. 박서연이 할일 목록 화면(S-03)에서 "기한초과" 필터를 활성화한다. (UC-405)
7. 시스템이 상태가 '완료'가 아니고 종료일자가 오늘 이전인 할일만 표시한다.

**결과 (Outcome)**
"거래처 계약서 검토" 할일이 기한초과 항목으로 표시되어 박서연이 미처리 업무를 즉시 파악할 수 있다.

### US-04. 내 정보 개인화

**페르소나:** P-02 김민준

**상황 (Context)**
김민준은 야간에 주로 앱을 사용하므로 다크 모드로 변경하고, UI 언어를 영어로 전환하고 싶다. 설정이 재로그인 후에도 유지되는지 확인하려 한다.

**목표 (Goal)**
테마를 dark로, 언어를 en으로 변경하고, 로그아웃 후 재로그인 시 설정이 자동 적용되는지 확인한다.

**시나리오 흐름**

1. 김민준이 내 정보 수정 화면(S-07)으로 이동한다.
2. 김민준이 테마 선택을 "dark"로 변경한다. (UC-203)
3. 시스템이 즉시 UI 전체를 다크 모드로 전환하고, 선택값을 DB에 저장한다. (UC-505)
4. 김민준이 언어 선택을 "en"으로 변경한다. (UC-204)
5. 시스템이 즉시 UI 전체 텍스트를 영어로 전환하고, 선택값을 DB에 저장한다. (UC-503)
6. 김민준이 저장 버튼을 클릭하여 변경 사항을 확정한다.
7. 김민준이 로그아웃한다. (클라이언트 측 JWT 토큰 삭제)
8. 김민준이 다시 로그인한다. (UC-103)
9. 시스템이 DB에 저장된 테마(dark)·언어(en)를 로드하여 자동 적용한다. (UC-205)

**결과 (Outcome)**
재로그인 후에도 다크 모드와 영어 UI가 자동으로 적용되어 김민준의 설정이 유지된다.

### US-05. 할일 수정 및 완료 처리

**페르소나:** P-01 이지수

**상황 (Context)**
이지수가 등록해 둔 "중간고사 공부" 할일의 종료일자를 추가하고, 공부를 마친 후 상태를 '완료'로 변경하려 한다.

**목표 (Goal)**
기존 할일에 종료일자를 추가하고 상태를 '완료'로 수정한다.

**시나리오 흐름**

1. 이지수가 할일 목록 화면(S-03)에서 "중간고사 공부" 항목을 선택하여 할일 수정 화면(S-05)으로 이동한다.
2. 이지수가 종료일자(2026-05-30)를 입력한다.
3. 이지수가 상태를 '완료'로 변경한다. (UC-407)
4. 시스템이 종료일자가 시작일자(미설정이므로 제약 없음)보다 이전이 아님을 확인한다. (UC-403)
5. 이지수가 저장한다.
6. 시스템이 할일을 수정하고 할일 목록 화면(S-03)으로 돌아간다.
7. 시스템이 상태가 '완료'이므로 기한초과 여부를 false로 계산한다. (UC-406)

**결과 (Outcome)**
"중간고사 공부" 할일이 '완료' 상태로 변경되고, 기한초과 대상에서 제외된다.

### US-06. 카테고리 삭제 시 할일 이동

**페르소나:** P-02 김민준

**상황 (Context)**
김민준이 더 이상 사용하지 않는 "임시" 카테고리를 삭제하려 한다. 해당 카테고리에는 할일이 남아 있다.

**목표 (Goal)**
"임시" 카테고리를 삭제하고, 해당 카테고리의 할일이 '기본' 카테고리로 자동 이동되었는지 확인한다.

**시나리오 흐름**

1. 김민준이 카테고리 관리 화면(S-06)으로 이동한다.
2. 김민준이 "임시" 카테고리의 삭제 버튼을 클릭한다. (UC-305)
3. 시스템이 "임시" 카테고리가 '기본' 카테고리가 아님을 확인한다. (UC-304)
4. 시스템이 "임시" 카테고리에 속한 모든 할일의 카테고리를 '기본'으로 변경한다. (UC-306)
5. 시스템이 "임시" 카테고리를 삭제하고 카테고리 목록을 갱신한다.
6. 김민준이 할일 목록 화면(S-03)에서 '기본' 카테고리 필터를 적용한다. (UC-405)
7. 시스템이 기존 "임시" 카테고리에 있던 할일들이 '기본' 카테고리 아래에 포함되어 표시한다.

**결과 (Outcome)**
"임시" 카테고리가 삭제되고, 해당 카테고리에 속해 있던 할일들은 데이터 손실 없이 '기본' 카테고리로 이동된다.
